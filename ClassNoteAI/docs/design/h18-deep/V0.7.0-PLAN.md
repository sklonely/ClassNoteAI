## 架構策略

### Token 系統：CSS Variables + Tailwind extend

```
H18_THEMES { light, dark } 
  ↓ 注入
:root + :root.dark { --h18-bg, --h18-text, --h18-accent, ... }
  ↓ Tailwind 讀
theme.extend.colors = { 'h18-bg': 'var(--h18-bg)', ... }
  ↓ 元件用
className="bg-h18-surface text-h18-text border-h18-border"
```

優點：theme 切換 = swap CSS class，所有元件**自動跟著切**，不需 prop 傳 T。

### Tailwind v.s. inline style：混合策略

- **Layout / 通用**：Tailwind classes
- **Theme-aware 動態值**：CSS vars + Tailwind arbitrary values（`text-[var(--h18-text)]`）或 inline style 讀 var
- **複雜動畫 / 計算值**：保留 inline style

不堅持 100% Tailwind。能用 class 就用 class，需要動態就 inline。

### 響應式策略

設計是固定 1440×900 letterbox。實際 app 視窗大小可變。對應策略：

- `min-width: 1200px, min-height: 700px` 維持（已有）
- 元件各自檢查 breakpoints（多數已有彈性 grid/flex）
- 真的不能縮的元件用 `min-w-` 鎖死，超出 scroll
- **不做平板/手機 layout** — Tauri 桌面 only

## 風險區與決策點

### R1 · Tailwind v3 動態值
Tailwind v3 部分 utility 不支援 `var()`。對策：用 arbitrary value 語法或保留 inline style。**先試試看，必要時升 Tailwind v4**（v4 改善了 var 支援）。

### R2 · 元件 prop drilling
原本元件不傳 T (theme)，因為用 Tailwind dark: prefix。新元件要不要也走 dark: pattern？**走** — 比 prop drilling theme 物件乾淨。

### R3 · AppSettings migration
新增 `appearance.{themeMode, density, fontSize, layout, toastStyle}` 欄位。舊 user 升級時要有 default。對策：
- storageService 讀 settings 時，缺欄位自動填 default
- 不需 DB migration script（純前端 JSON）

### R4 · 既有元件混用 Tailwind dark: 跟 H18 vars
過渡期會有元件還用 `dark:bg-slate-800` (舊)、有的用 `bg-h18-surface` (新)。對策：
- Phase 0 同時定義「舊→新」mapping table（PR 描述裡寫明）
- 每個 PR 標記它換了哪些元件
- 直到所有 component 都換完才升 beta

### R5 · 測試影響
- vitest 元件測試多用 className 斷言
- UI 大改後**測試斷言要全部審視**
- 對策：每個 PR 把對應 test snap update

### R6 · CI 影響
- Tailwind config 改可能影響 production build size
- CSS vars 在 IE 不支援 — 我們不支援 IE，OK
- WebView2 (Windows) / WebKit (Mac) 都支援 → OK

### R7 · 平台差異 (traffic lights)
- macOS 原生左上、Windows 原生右上、Linux 看 DE
- 設計用 macOS 風放左上 — 對 Windows 用戶會覺得「位置怪怪的」
- **決策：v0.7.0 alpha 全平台統一放左上**，理由：跨平台一致性 > 平台原生
- 若反饋強烈再加 OS-detect（v0.7.x patch）
