## 架構策略

### 樣式方案：CSS Modules + CSS Variables（**不用 Tailwind**）

決策過程見 `STYLING-DECISION.md`（如有）或 commit history。簡述：
- Tailwind v3→v4 的 framework drift 已咬過一次（v3 → v4 fidelity 問題）
- H18 設計大量非 scale 值 (14px, 22px, 0.18em 等)，Tailwind arbitrary
  syntax 反而醜
- CSS Modules 跟 prototype inline style 對應度 1:1
- CSS spec 不會 breaking change，5-10 年不再被框架咬
- Type safety 用 `typescript-plugin-css-modules` + `tokens.ts` 補足

### Token 系統：純 CSS Variables

```
H18_THEMES { light, dark }  (prototype 物件)
  ↓ 抄到
src/styles/tokens.css    :root { --h18-text: ... }   :root.dark { --h18-text: ... }
src/styles/tokens.ts     export const TOKENS = { color: { text: 'var(--h18-text)' } }
  ↓
.module.css 直接用 var(--h18-*)
.tsx 想動態取值用 TOKENS.color.text (TS type-safe)
  ↓
切 theme = document.documentElement.classList.toggle('dark')
所有元件自動跟，不需 prop drilling
```

### CSS Modules 寫法

```tsx
// Card.tsx
import s from './Card.module.css';
import clsx from 'clsx';

export function Card({ active, children }: Props) {
  return <div className={clsx(s.root, active && s.active)}>{children}</div>;
}
```

```css
/* Card.module.css */
.root {
  padding: 14px;
  border-radius: 8px;
  background: var(--h18-surface);
  border: 1px solid var(--h18-border);
}
.active {
  border-color: var(--h18-accent);
}
```

### Tailwind 移除策略

**不在 Phase 0 移除**。Tailwind 跟 CSS Modules 在 Vite 可以並存：
- 既有 28 元件**繼續用 Tailwind**
- 新元件**全用 CSS Modules**
- Phase 5 全部主頁面 port 完才刪 `tailwind.config.js` + `npm uninstall tailwindcss`

避免在 Phase 0 就 break 既有 28 元件。

### 響應式策略

設計是固定 1440×900 letterbox。實際 app 視窗大小可變。對應策略：

- `min-width: 1200px, min-height: 700px` 維持（已有）
- 元件各自檢查 breakpoints（多數已有彈性 grid/flex）
- 真的不能縮的元件用 `min-w-` 鎖死，超出 scroll
- **不做平板/手機 layout** — Tauri 桌面 only

## Development Methodology · TDD

每個元件 / utility / hook 採 **Red-Green-Refactor TDD**：

```
1. 紅 — 寫 .test.ts(x) 列出預期行為，部分 failing
2. 綠 — 寫實作直到 tests pass
3. 重構 + 視覺對齊 — 開 prototype html 對照，微調 CSS module 細節
4. Commit — 一個元件 / hook 一個 commit
```

### TDD 能抓的 vs 不能抓的

✅ **能**：行為契約、props、state 切換、事件處理、Manager pub/sub、
鍵盤快捷鍵、持久化、hook 卸載清理、type contracts

❌ **不能**：視覺像不像 H18、動畫流暢度、實際色彩對比、hover 視覺
feedback、responsive layout（jsdom 沒 layout）

→ 視覺對齊靠**眼睛 + prototype 對照**（每個 PR description 列出對照位置）

### 工具（已在 repo）

```
vitest + @testing-library/react + @testing-library/user-event
+ @testing-library/jest-dom + jsdom + @vitest/coverage-v8
```

### 覆蓋率目標（v0.7.0-alpha 期間不設 CI gate）

- Manager / utils / hooks: **90%+**
- Component: **70%+**（行為導向不追 line coverage）
- 不寫 snapshot tests（brittle）
- 不寫 E2E（之後）
- 不寫 visual regression（之後）

## 風險區與決策點

### R1 · Tailwind v3 動態值
Tailwind v3 部分 utility 不支援 `var()`。對策：用 arbitrary value 語法或保留 inline style。**先試試看，必要時升 Tailwind v4**（v4 改善了 var 支援）。

### R2 · 元件 prop drilling
原本元件不傳 T (theme)，因為用 Tailwind dark: prefix。新元件要不要也走 dark: pattern？**走** — 比 prop drilling theme 物件乾淨。

### R3 · AppSettings migration
新增 `appearance.{themeMode, density, fontSize, layout, toastStyle}` 欄位。舊 user 升級時要有 default。對策：
- storageService 讀 settings 時，缺欄位自動填 default
- 不需 DB migration script（純前端 JSON）

### R4 · 既有元件混用 Tailwind dark: 跟 H18 vars
過渡期會有元件還用 `dark:bg-slate-800` (舊)、有的用 `bg-h18-surface` (新)。對策：
- Phase 0 同時定義「舊→新」mapping table（PR 描述裡寫明）
- 每個 PR 標記它換了哪些元件
- 直到所有 component 都換完才升 beta

### R5 · 測試影響
- vitest 元件測試多用 className 斷言
- UI 大改後**測試斷言要全部審視**
- 對策：每個 PR 把對應 test snap update

### R6 · CI 影響
- Tailwind config 改可能影響 production build size
- CSS vars 在 IE 不支援 — 我們不支援 IE，OK
- WebView2 (Windows) / WebKit (Mac) 都支援 → OK

### R7 · 平台差異 (traffic lights)
- macOS 原生左上、Windows 原生右上、Linux 看 DE
- 設計用 macOS 風放左上 — 對 Windows 用戶會覺得「位置怪怪的」
- **決策：v0.7.0 alpha 全平台統一放左上**，理由：跨平台一致性 > 平台原生
- 若反饋強烈再加 OS-detect（v0.7.x patch）

### Soft CPs · 我會主動停下問你

不是 phase 結束才停，而是遇到**判斷題**就停：

- **架構決策**：計畫沒覆蓋的設計問題
  （例：token 命名、新 component 該歸在 h18/ 還是 別處）
- **Scope creep**：某 phase 實際比預期大很多，可能影響後續排程
- **不可逆動作**：force push、刪檔、改 git history、刪 dependency
