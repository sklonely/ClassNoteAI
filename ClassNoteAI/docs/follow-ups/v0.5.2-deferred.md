## F1 — Silero VAD (replace energy-threshold VAD)

**Priority:** P1 (quality)
**Effort:** ~1 day
**Risk:** medium — adds ONNX runtime, possible conflict with ct2rs

### Problem

`transcriptionService.ts` currently uses a 0.002 energy threshold
(`CONFIG.VAD_ENERGY_THRESHOLD`). Misfires on:
- HVAC rumble, projector fan, hallway chatter bleeding through
- Keyboard clicks, chair scrapes
- Prof clearing throat
- Background students coughing / whispering

[Silero VAD](https://github.com/snakers4/silero-vad) is a 1.8 MB
neural VAD, 1 ms per 30 ms chunk on CPU, 100+ languages.

### Why not done overnight

`service.rs` has a comment explicitly noting protobuf conflict
concerns with ct2rs (our translation backend). Adding `ort` to the
workspace may break the Windows build — the PR-check CI would catch
it, but recovering from a broken build mid-night isn't something I
was willing to risk without the user watching.

### Implementation sketch

1. Add `ort = "2"` to `src-tauri/Cargo.toml`. Verify no feature-flag
   conflict with `ct2rs` on Windows + macOS locally.
2. Download `silero_vad.onnx` (1.8 MB) via
   `EmbeddingModelConfig::silero_vad()` in `download.rs`.
3. Wrap inference in `src-tauri/src/vad/silero.rs`:
   ```rust
   pub struct SileroVad { session: ort::Session, state: [f32; 128] }
   impl SileroVad {
       pub fn is_speech(&mut self, samples: &[i16]) -> f32 { … }
   }
   ```
4. Tauri command `vad_check_silero(audio_chunk)` that transcriptionService
   calls instead of / alongside energy threshold.
5. Golden-audio fixtures under `evals/fixtures/vad/` for accuracy regression.

### Alternative if ort conflicts with ct2rs

Wrap Silero via `candle` directly (there's a community Candle-silero
implementation). Slightly more work but avoids a new runtime.

## F2 — Hybrid BM25 + dense + cross-encoder reranker

**Priority:** P2 (retrieval quality)
**Effort:** ~3-5 days
**Risk:** medium-high — large architectural change to RAG

### Problem

`ragService.chat` currently does pure cosine top-K over bge-small-en
embeddings. 2026 production-RAG [standard](https://superlinked.com/vectorhub/articles/optimizing-rag-with-hybrid-search-reranking)
is:
1. Hybrid fan-out: BM25 (keyword) + dense (semantic) in parallel.
2. Merge with Reciprocal Rank Fusion.
3. Cross-encoder rerank of top-50 → top-5.

Measured gain: +17 pp MRR@3, +12 pp Recall@5 over pure dense.

### Why not done overnight

Large change to the retrieval pipeline. Needs:
- A BM25 implementation (Rust `tantivy` crate OR JS `minisearch`)
- Another ~550 MB model on disk for BGE Reranker v2-m3
- Care around the cross-lingual story: should BM25 run on the
  English-translated query or the original Chinese?

### Implementation sketch

1. **BM25 side**: add `minisearch` to frontend, index chunks at the
   same time as embeddings are stored. Alternative: Rust `tantivy`
   via a Tauri command — faster queries but another native compile
   target for Windows CI.
2. **Fusion**: RRF with `k = 60`. `(1/(60 + rankBM25)) + (1/(60 + rankDense))`.
3. **Reranker**: Candle + BAAI/bge-reranker-v2-m3. Standard BERT so
   Candle handles it.
4. **Cross-lingual**: for CJK queries, run BM25 on the translated
   English form (matches the existing dense path). Adds no extra
   translation calls.

### Gotchas

- BM25 index gets out-of-sync with embeddings table on chunk edits.
  Needs a shared rebuild path.
- Reranker is ~550 MB — doubles the on-disk model footprint. May
  want a settings toggle.

## F3 — sqlite-vec HNSW index

**Priority:** P3 (scale)
**Effort:** ~2-3 days
**Risk:** high — native extension, per-platform compilation

### Problem

`embeddingStorageService.semanticSearch` does in-process O(n) cosine
over all chunks for a lecture. Fine at current corpus size (a few
thousand chunks per lecture). Doesn't scale to course-level search
across dozens of lectures.

### Why not done overnight

[sqlite-vec](https://github.com/asg017/sqlite-vec) is a native
SQLite extension. Needs to be compiled for each target in our
release matrix (Windows x64, macOS ARM, macOS x64). A build-system
change at 3am, with no one watching, is exactly the kind of thing
that breaks installers silently.

### Implementation sketch

1. Add `sqlite-vec` as a vendored dependency.
2. Update `src-tauri/scripts/bootstrap-vendor.sh` to pull the right
   binary for the current OS/arch.
3. Migration in `database.rs` to convert the existing `embeddings`
   table into a `vec0(embedding FLOAT[384])` virtual table.
4. Wire release workflows (`.github/workflows/release-*.yml`) to
   bundle the extension binary.

### Blocker

None that I can see from static review — just needs a person
watching the release build come out green on both platforms.

## F4 — Topic-based section boundaries

**Priority:** P2 (summary quality)
**Effort:** ~2 days
**Risk:** low — no new deps

### Problem

`NotesView.handleStopRecording` currently segments the lecture into
hard-coded 5-minute sections
([`NotesView.tsx` auto-note generation](../../src/components/NotesView.tsx#L788)).
No correlation with actual topic shifts — sections often cut
mid-concept.

### Approach

Two options, cheaper first:

1. **PDF-pinned sections**: PDF turn-page events already carry
   timestamps if the user was scrolling along. Use those as
   section boundaries.
2. **Embedding similarity dip detection**: compute embedding for
   each 1-minute window, flag boundaries where adjacent-window
   cosine drops below a threshold. Off-the-shelf algorithm
   (TextTiling-style).

### Why not done overnight

Requires evaluation — a bad auto-segmentation is worse than no
auto-segmentation. Should be behind a settings toggle and measured
with user feedback before replacing the current 5-min default.

## F6 — Fill in the eval harness

**Priority:** P2
**Effort:** ~2-3 days
**Risk:** low

`evals/scripts/asr-wer.ts` and `rag-mrr.ts` have the metric math
locked down with unit tests, but the actual ASR / retrieval
invocations are stubbed. Wiring needs:

- **ASR**: either a standalone `whisper.cpp` binary invoked via
  `child_process`, OR a small Rust crate exposing whisper-rs via
  stdin/stdout so `tsx` can drive it.
- **RAG**: a headless retrieval harness. `generateLocalEmbedding`
  requires the Tauri runtime today; either (a) extract a Node-
  compatible embedding path using a WASM-compiled bge, or (b)
  spawn the Tauri test binary in CI.

Once wired, add golden fixtures to `evals/fixtures/asr/` and
`evals/fixtures/rag/` and the nightly workflow starts producing
real numbers.
